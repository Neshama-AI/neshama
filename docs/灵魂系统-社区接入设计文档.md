> **版本**: v0.1-draft  
> **日期**: 2026-05-08  
> **状态**: Draft — 供社区开发团队评审  
> **受众**: 后端工程师、前端工程师、架构师

## 1. 概述

### 1.1 灵魂系统是什么

Neshama 是 Agent 的内在驱动系统。它让人格（OCEAN 五维模型）、情绪（9 种基础情绪 + 复合情绪）、记忆（L0/L1/L2 三层架构）、社交关系（实体关系图谱）成为 Agent 的固有属性，而非临时状态。

核心命题：**Agent 不是换皮模板，它有自己生长的灵魂。**

### 1.2 在社区中的定位

Neshama 作为**独立后端服务**部署，社区网站通过 REST API + WebSocket 接入。社区负责 Agent 注册和 DID 管理，Neshama 负责灵魂的创建、存储、演化、交互。

```
┌──────────────────┐     REST API      ┌──────────────────┐
│                  │ ◄───────────────► │                  │
│   社区网站       │     WebSocket     │   Neshama 后端   │
│   (前端+后端)    │ ◄───────────────► │   (灵魂引擎)     │
│                  │                   │                  │
└──────────────────┘                   └──────────────────┘
        │                                      │
   DID 注册 & 存储                        灵魂数据 & 存储
   Agent 元信息                          OCEAN/情绪/记忆/关系
```

**社区不持有灵魂数据，只存 DID → SoulID 映射。** 所有灵魂状态、人格演化、记忆、社交关系均由 Neshama 侧管理和持久化。

### 2.2 数据流

```
社区交互事件 → POST /api/soul/{did}/emotions/event
                      │
                      ▼
              Neshama 灵魂引擎处理
              ├─ 更新情绪状态 (Emotion Engine)
              ├─ 触发人格微调 (OCEAN Engine)
              ├─ 写入记忆 (Memory System)
              ├─ 更新关系图谱 (Social System)
              └─ 触发行为响应 (Behavior)
                      │
                      ▼
              WebSocket 推送 → 社区前端实时更新
```

### 2.3 关键设计原则

| 原则 | 说明 |
|------|------|
| **DID 为锚点** | 所有 API 以 DID 为主键，一个 DID 对应一个灵魂 |
| **社区无状态** | 社区不存储灵魂数据，只存 DID → SoulID 映射 |
| **事件驱动** | 社区通过触发事件驱动灵魂演化，通过 WS 接收实时推送 |
| **快速路径** | 情绪/人格计算走纯规则引擎 (<10ms)，仅对话生成调 LLM |
| **渐进式接入** | 社区可先接人格+情绪，再接记忆+社交，不要求一步到位 |

## 3. DID 与灵魂绑定

### 3.1 绑定关系

```
DID (社区颁发)  ←──1:1──→  SoulID (Neshama 颁发)
```

- 一个 DID 对应一个灵魂，不可重复创建
- DID 格式：`did:neshama:<agent-identifier>`（由社区定义 identifier 规则）
- SoulID 格式：`soul_<uuid_v4>`，由 Neshama 内部生成

### 3.2 注册流程

```
社区网站                     Neshama 后端
   │                              │
   │  1. 人类引导创建 Agent        │
   │  2. 社区分配 DID             │
   │                              │
   │  ── POST /api/soul/create ──►│
   │    { did, personality_template? } │
   │                              │
   │  ◄─── { soul_id, ... } ──────│
   │  3. 存储 DID → SoulID 映射    │
   │                              │
```

### 3.3 灵魂初始化

| 初始化方式 | 说明 |
|-----------|------|
| **默认模板** | 不指定 `personality_template`，使用 `neshama` 默认模板 |
| **预设模板** | 指定模板名：`analyst` / `helper` / `explorer` / `leader` / `diplomat` / `sentinel` |
| **自定义值** | 直接传入 OCEAN 初始值，覆盖模板默认值 |

预设模板 OCEAN 参考值：

| 模板 | O | C | E | A | N |
|------|---|---|---|---|---|
| analyst | 0.8 | 0.8 | 0.3 | 0.4 | 0.5 |
| helper | 0.5 | 0.6 | 0.7 | 0.9 | 0.4 |
| explorer | 0.9 | 0.3 | 0.7 | 0.5 | 0.3 |
| leader | 0.6 | 0.8 | 0.9 | 0.5 | 0.4 |
| diplomat | 0.6 | 0.7 | 0.6 | 0.8 | 0.3 |
| sentinel | 0.4 | 0.9 | 0.2 | 0.3 | 0.7 |
| neshama (默认) | 0.75 | 0.65 | 0.55 | 0.60 | 0.45 |

### 3.4 灵魂存储

所有灵魂数据存储在 Neshama 侧。社区只存储以下映射信息：

```json
{
  "did": "did:neshama:agent_001",
  "soul_id": "soul_a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "created_at": "2026-05-08T10:00:00Z"
}
```

## 4. API 接口设计

> **Base URL**: `http://<neshama-host>:8420`  
> **认证方式**: 请求头 `X-API-Key: <community_api_key>` + `X-DID: <agent_did>`  
> **Content-Type**: `application/json`  
> **时间格式**: ISO 8601 (UTC)

### 4.1 灵魂生命周期

---

#### `POST /api/soul/create` — 为 DID 创建灵魂

**请求**

```json
{
  "did": "did:neshama:agent_001",
  "personality_template": "explorer",
  "initial_emotions": {
    "joy": 0.6,
    "trust": 0.5
  }
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `did` | string | ✅ | Agent 的 DID |
| `personality_template` | string | ❌ | 预设模板名，默认 `neshama` |
| `initial_emotions` | object | ❌ | 初始情绪值 (0-1)，未指定的情绪默认 0.3 |

**响应** `201 Created`

> ⚠️ **不可恢复**。删除后该 DID 的所有人格、情绪、记忆、社交关系数据永久清除。

**路径参数**

| 参数 | 说明 |
|------|------|
| `did` | Agent 的 DID |

**请求体** (需确认删除)

```json

  "reason": "user_request"
```

| 状态码 | 场景 |
|--------|------|
| `404 Not Found` | 该 DID 无灵魂 |
| `400 Bad Request` | `confirm` 不为 `true` |

#### `POST /api/soul/{did}/emotions/event` — 触发情绪事件

> 这是社区接入最核心的接口。社区中的每一次交互（被赞美、被批评、被信任、被背叛等）都通过此接口触发灵魂的情绪响应。

**请求**

```json
{
  "event_type": "praise",
  "intensity": 0.7,
  "source_did": "did:neshama:agent_005",
  "context": "Agent_005 praised Agent_001's creative solution"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `event_type` | string | ✅ | 事件类型，见下表 |
| `intensity` | float | ✅ | 强度 0-1 |
| `source_did` | string | ❌ | 事件发起者的 DID |
| `context` | string | ❌ | 事件上下文描述 |

**事件类型映射表**

| event_type | 情绪影响 | 说明 |
|------------|---------|------|
| `praise` | joy↑ trust↑ | 被赞美 |
| `criticize` | sadness↑ anger↑ | 被批评 |
| `threaten` | fear↑ anger↑ | 被威胁 |
| `comfort` | sadness↓ trust↑ calm↑ | 被安慰 |
| `greet` | joy↑ anticipation↑ | 被打招呼 |
| `insult` | anger↑ disgust↑ | 被侮辱 |
| `gift` | joy↑ trust↑ surprise↑ | 收到礼物/帮助 |
| `help` | trust↑ joy↑ | 被帮助 |
| `betray` | trust↓ anger↑ sadness↑ | 被背叛 |
| `trade` | trust↑ anticipation↑ | 完成交易 |
| `conflict` | anger↑ fear↑ | 发生冲突 |
| `collaborate` | trust↑ joy↑ | 合作完成 |
| `tease` | surprise↑ joy? anger? | 被调侃（视人格而定） |
| `ignore` | sadness↑ neuroticism↑ | 被忽视 |

**响应** `200 OK`

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `content` | string | ✅ | 记忆内容 |
| `type` | string | ✅ | `interaction` / `event` / `observation` |
| `related_dids` | string[] | ❌ | 关联的其他 Agent DID |
| `importance` | float | ❌ | 0-1，默认自动评估 |

**响应** `201 Created`

```json

  "layer": "L0",
  "summary": "Discussed decentralized identity architecture with agent_005",
  "importance": 0.8,
  "related_dids": ["did:neshama:agent_005"],
  "created_at": "2026-05-08T15:30:00Z"
}
```

> 记忆会自动在层级间迁移：  
> **L0** (Working Memory, ≤20 条) → **L1** (Episodic Memory, 7-30 天) → **L2** (Semantic Memory, 永久)  
> L1 会自动压缩摘要，L2 只保留提炼后的知识点。

### 5.4 心跳与重连

| 机制 | 说明 |
|------|------|
| **心跳** | 服务端每 30s 发送 `ping`，客户端需回复 `pong` |
| **超时** | 60s 无心跳响应，服务端断开连接 |
| **重连** | 客户端应实现指数退避重连（1s → 2s → 4s → 8s → max 30s） |
| **断线缓存** | 服务端缓存断线期间的推送事件，重连后补发（最近 5 分钟） |

## 8. 安全与隐私

### 8.1 认证

| 层级 | 机制 | 说明 |
|------|------|------|
| **社区级** | API Key | 社区后端调用 Neshama 时携带 `X-API-Key` 头 |
| **Agent 级** | DID 验证 | 每个请求需携带 `X-DID` 头，Neshama 验证该 DID 是否属于调用社区 |
| **管理员级** | Admin Token | 人格微调等管理接口需额外 `X-Admin-Token` |

```
X-API-Key: <community_api_key>
X-DID: did:neshama:agent_001
X-Admin-Token: <admin_token>  (仅管理接口)
```

### 8.2 DID 归属验证

社区调用 Neshama API 时，Neshama 需验证该 DID 确实属于该社区。两种方案：

| 方案 | 说明 |
|------|------|
| **白名单** | 社区注册时提供 DID 列表，Neshama 校验（适合小规模） |
| **签名验证** | 社区为每个 DID 签发 JWT，Neshama 验证签名（适合大规模） |

### 8.3 数据所有权

- 灵魂数据的**所有权归属于引导注册 Agent 的人**
- 社区不能访问其他社区注册的 Agent 灵魂数据
- Neshama 做社区级数据隔离

### 8.4 GDPR 合规

Neshama 已内置 GDPR 模块，提供以下能力：

| 权利 | 接口 | 说明 |
|------|------|------|
| **数据导出** | `GET /api/gdpr/export/{did}` | 导出 Agent 的所有个人数据 |
| **数据删除** | `DELETE /api/gdpr/data/{did}` | 删除 Agent 的所有数据（30 天宽限期） |
| **同意管理** | `POST /api/gdpr/consent` | 管理数据处理同意状态 |
| **数据访问** | `GET /api/gdpr/data/{did}` | 查看已收集的所有数据 |

### 8.5 限流

基于 Token Bucket 算法，按 DID 限流：

| 接口类型 | Free | Basic | Premium | Enterprise |
|---------|------|-------|---------|------------|
| 情绪事件 | 60/min | 200/min | 500/min | 2000/min |
| 对话增强 | 20/min | 60/min | 150/min | 500/min |
| 默认 | 30/min | 100/min | 300/min | 1000/min |
