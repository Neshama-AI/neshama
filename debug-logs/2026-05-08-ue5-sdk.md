  1. 日志唯一Error在uhtmanifest(1)行，说明UHT阶段即失败，0个文件生成

  3. 这是UE5内置OnlineFramework的社交系统类型定义，与我们的NPC社交系统类型文件同名
  4. UHT按文件basename去重，不区分目录层级，因此即使路径不同也会冲突
- 修复：
  1. `SocialTypes.h` → `NeshamaSocialTypes.h`（加Neshama前缀消除冲突）
  2. `SocialTypes.cpp` → `NeshamaSocialTypes.cpp`
  3. 更新3处include引用（generated.h、cpp、SocialEngine.h）
- 涉及文件：3个文件修改/重命名
- 类型名检查：`ESocialInteractionType`和`ERelationshipCategory`与引擎不冲突，无需改名
- 主动扫描：检查所有SDK头文件名是否还有潜在冲突（SocialEngine.h/EntityGraph.h/MemorySystem.h等），均为LOW风险
- Commit: 6890e60, 打包v12

### R15 — R14残留编译错误修复 (22:30)

#### 错误分析
用户用命令行Build.bat编译，40个编译单元，12类去重错误。大部分是R14修复不完整或修复方式有误。

#### 逐错误诊断

1. **EmotionEngine.cpp(7)**: C1083 "SoulEngine/Utils/SoulMathUtils.h"不存在
   - R14修了路径但include还是旧的`SoulEngine/Utils/`前缀
   - 修复: → `Utils/SoulMathUtils.h`

2. **LicenseManager.cpp(7)**: C1083 "Misc/SHA.h"不存在
   - R14将`Misc/SecureHash.h`改成`Misc/SHA.h`，但UE5.6.1中此头文件不存在！
   - R14的TOOLS.md记录"FSHA256Hash→FSHA256（在Misc/SHA.h非Misc/SecureHash.h）"是错误的
   - 修复: 改回`Misc/SecureHash.h`，FSHA256→FSHA1（FSHA256在UE5.6.1中不可用）
   - 注意: FSHA1是20字节哈希，对机器指纹够用

3. **NPCSoulComponent.cpp(101)**: C2100 对bool解引用 `*IsConnected()`
   - `*`是多余的，IsConnected()返回bool不是指针
   - 修复: 移除`*`

4. **NeshamaConfig.cpp(249/272/273/296)**: C2668 EscapeJsonString重载歧义
   - ADL（参数相关查找）: 因为参数是FString，编译器在FNeshamaUtils命名空间和UE全局命名空间都找到了同名函数
   - UE引擎的JsonWriter.h(71)有`inline FString EscapeJsonString(const FString& StringVal)`
   - 修复: 所有调用加`FNeshamaUtils::`前缀限定

5. **NeshamaClient.cpp(154/198)**: C2672 BindLambda签名不匹配
   - Lambda参数`(IHttpRequest*, IHttpResponse*, bool)`但委托期望`(FHttpRequestPtr, FHttpResponsePtr, bool)`
   - UE5.6: `FHttpRequestPtr = TSharedRef<IHttpRequest, ESPMode::ThreadSafe>`, `FHttpResponsePtr = TSharedPtr<IHttpResponse, ESPMode::ThreadSafe>`
   - 修复: 改lambda参数类型 + `InResponse`用`.IsValid()`而非`!= nullptr`

6-9. **Editor模块**: FEditorStyle→FAppStyle, SExpanderArrow无HeaderContent, 缺方法声明, STextBlock无Padding, SVerticalBox::Slot无Visibility, SComboBox签名不匹配, SButton::OnClicked不支持三元表达式
   - 这些都是UE5.6 Slate API breaking changes
   - SExpanderArrow在UE5.6中仅用于TreeView行，其FArguments只有IndentAmount/ShouldDrawWires
   - STextBlock的Padding属性在UE5.6已移除
   - SVerticalBox::FSlot不再有Visibility属性
   - SComboBox<TSharedPtr<FString>>::OnSelectionChanged签名需要TSharedPtr<FString>而非FString
   - SButton::OnClicked宏不支持三元表达式选择函数指针

- TRawPtrProxy没有operator->，所以->SetStringField报错
- UE5.6 MakeShareable返回中间代理类型，必须显式声明目标智能指针类型
- 修复：`TSharedPtr<FJsonObject> RequestBody = MakeShareable(new FJsonObject())`

**E5: SNeshamaSetupWizard OnClicked三元 (C2446 + C2661 + C7732 + C2679)**

### R17 Debug Notes

#### 输入日志: `./用户上传/报错_1778251390539_0_ytmc.txt` (39行)
#### 错误统计: 3个error C2664，1个文件（LicenseManager.cpp），1类去重

#### 逐类调试

**E1: MakeLicenseRequest参数类型不匹配 (C2664 × 3)**
- 错误行: LicenseManager.cpp(140/210/252)
- 函数签名: `MakeLicenseRequest(const FString&, const TSharedRef<FJsonObject>&, TFunction<...>)`
- 调用传入: `TSharedPtr<FJsonObject> RequestBody`
- `TSharedPtr` 不能隐式转为 `TSharedRef`（前者可空后者不可空，语义不同）
- 根因追溯: R16将 `auto RequestBody` → `TSharedPtr<FJsonObject> RequestBody` 修正了auto陷阱，但未同步修改函数参数类型
- 修复: 将 `MakeLicenseRequest` 参数2从 `TSharedRef` → `TSharedPtr`（.h + .cpp同步修改）
- 防御: 在函数开头添加 `if (!RequestBody.IsValid())` null check + early return

#### 经验教训
- 修改变量类型时必须检查函数签名是否匹配
- `TSharedRef` → `TSharedPtr` 降级需要添加null check
