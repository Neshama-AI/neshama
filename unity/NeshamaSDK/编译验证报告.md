## 发现并修复的问题

### 🔴 严重编译错误（已修复）

#### 1. BehaviorHint 缺少 suggested_actions 字段
- **文件**: `Runtime/Models/BehaviorHint.cs`
- **问题**: `NPCSoul.IsWillingToTalk()` 和 `BehaviorResponse.ShouldRefuseConversation()` 引用了 `behavior.suggested_actions`，但 `BehaviorHint` 类没有此字段
- **修复**: 添加 `public List<string> suggested_actions` 字段

#### 2. NeshamaConfig 不能作为 ScriptableObject 使用
- **文件**: `Runtime/NeshamaConfig.cs`, `Editor/NeshamaSettingsProvider.cs`
- **问题**: `NeshamaSettingsProvider` 调用 `ScriptableObject.CreateInstance<NeshamaConfig>()`，但 `NeshamaConfig` 是普通 `[Serializable]` 类
- **修复**: 将 `NeshamaConfig` 改为继承 `ScriptableObject`，并修改 `CreateDefault()` 使用 `ScriptableObject.CreateInstance`

#### 3. NeshamaClient.PostAsync 是 private 方法
- **文件**: `Runtime/NeshamaClient.cs`, `Editor/SetupWizard.cs`
- **问题**: `SetupWizard` 调用 `client.PostAsync<object>("/api/auth/...", ...)`，但该方法为 `private`
- **修复**: 将 `PostAsync` 改为 `internal`，并新增 `PostAbsoluteAsync` 和 `GetAbsoluteAsync` 方法用于非 game 前缀的端点

#### 4. Auth 端点路径错误
- **文件**: `Editor/SetupWizard.cs`
- **问题**: 调用 `/api/auth/register` 等路径时，`BuildUrl` 会加上 `/api/game` 前缀，导致实际路径变成 `/api/game/api/auth/register`
- **修复**: 使用 `PostAbsoluteAsync` 代替 `PostAsync`，直接使用完整路径

#### 5. CreateNPC API 路径末尾多余斜杠
- **文件**: `Runtime/NeshamaClient.cs`
- **问题**: `CreateNPCAsync` 使用 `/npc/` 路径（带尾部斜杠），后端实际为 `/npc`
- **修复**: 移除尾部斜杠

#### 6. Sample 代码在运行时使用 UnityEditor.SerializedObject
- **文件**: `Samples/TavernDemo/Scripts/TavernKeeperAI.cs`, `GuardCaptainAI.cs`
- **问题**: 运行时代码使用 `new SerializedObject(npcSoul)`，这只能在 Editor 环境中编译，不能打包
- **修复**: 在 `NPCSoul` 中添加 `Configure()` 公共方法，Sample 改用此方法

#### 7. Sample 代码缺少 TMPro using 引用
- **文件**: `Samples/TavernDemo/Scripts/TavernKeeperAI.cs`, `GuardCaptainAI.cs`
- **问题**: 使用了 `TMPro.TextMeshProUGUI` 和 `TextAlignmentOptions`，但没有 `using TMPro;`
- **修复**: 添加 `using TMPro;` 引用

### 🟡 运行时错误（已修复）

#### 8. API 响应格式不匹配
- **文件**: `Runtime/NeshamaClient.cs`
- **问题**: 后端所有响应使用 `{"success": true, "data": {...}}` 包装格式，但 SDK 直接反序列化为业务模型，导致所有字段为空
- **修复**: 
  - 新增 `Runtime/Models/ApiResponse.cs`，包含所有 API 的响应包装器类型
  - 更新 `NeshamaClient` 中所有 API 调用，先反序列化为包装器类型，再提取 `data` 字段

#### 9. JsonUtility 不支持 Dictionary 序列化
- **文件**: `Runtime/Models/EmotionState.cs`, `GameEvent.cs`, `NPCProfile.cs`, `RelationGraph.cs`
- **问题**: Unity 的 `JsonUtility` 不支持 `Dictionary<string, float>` 和 `Dictionary<string, object>` 序列化/反序列化
- **修复**:
  - `EmotionState.emotions`: 改为 `List<EmotionEntry>` + 懒加载缓存 Dictionary
  - `GameEvent.context`: 改为 `List<ContextEntry>` + 属性访问器
  - `NPCProfile.metadata`, `CreateNPCRequest.config`, `RelationEntity.attributes`: 标记为 `[System.NonSerialized]`

### 🟢 已修复的改进

#### 10. SetupWizard 中也使用 SerializedObject
- **文件**: `Editor/SetupWizard.cs`
- **问题**: `CreateNPCGameObject()` 使用 `SerializedObject` 配置 NPC，与 Sample 不一致
- **修复**: 改用 `NPCSoul.Configure()` 方法

## 剩余警告（非编译错误）

### ⚠️ 1. Sample 对 TMPro 的硬依赖
- **文件**: `Samples/TavernDemo/Scripts/TavernKeeperAI.cs`, `GuardCaptainAI.cs`
- **说明**: Sample 直接引用 `TMPro.TextMeshProUGUI`，如果项目未安装 TextMeshPro 包将编译失败
- **建议**: 在 Sample 文件头部添加注释说明 TMP 依赖，或用 `#if TEXTMESH_PRO_PRESENT` 条件编译包裹

### ⚠️ 2. 后端 API 参数格式差异（运行时兼容性）
- **说明**: 后端部分端点使用 query parameters 而非 JSON body：
  - `POST /npc` → 后端接受 `name`, `personality`, `preset`, `npc_id` 作为 query params
  - `POST /npc/{id}/chat` → 后端接受 `message` 作为 query param
  - `POST /npc/{id}/event` → 后端接受 `event_type`, `intensity`, `context` 作为 query params
  - `POST /npc/{id}/remember` → 后端接受 `entity_id`, `relation_type`, `weight` 作为 query params
- SDK 当前以 JSON body 方式发送这些参数
- **影响**: 当前后端的 FastAPI 配置可能同时支持 query params 和 body，但建议后续统一格式
- **建议**: 在后端添加 Pydantic Request Body 模型，统一接受 JSON body
