# Seele 机制完善方案

## 日期
2026-05-08

## 背景
Seele 在 UE5 编译验证过程中频繁截断（R1~R8），暴露了5个机制缺陷。用户要求全部解决，本方案逐项设计并给出具体可执行的修改内容。

---

## 问题清单与方案

### P1. 任务模板标准化

**问题**：每次 `sessions_spawn` 的任务描述格式不统一，容易遗漏关键信息（上下文、文件路径、角色身份、禁忌等），导致 sub-agent 执行偏差。

**方案**：在 SOUL.md 新增强制任务模板，所有 spawn 调用必须套用。

**模板内容**（追加到 SOUL.md）：
```markdown
## 任务模板（sessions_spawn 强制使用）

每次 sessions_spawn 的任务描述必须包含以下 5 段，缺一不可：

### 身份段（必填）
你现在是{花名}（{角色}），负责{职责}。禁忌：{禁忌}

### 任务段（必填）
{具体任务描述，1-3句，说清做什么}

### 上下文段（必填）
- 项目：{项目名}
- 相关文件：{路径列表，至少1个}
- 前置状态：{已完成什么/卡在哪}
- 依赖信息：{需要知道的外部事实}

### 产出段（必填）
- 输出文件：{路径}
- 完成后写 dev-log 到 ./Neshama/dev-logs/
- 需要回写的经验 → ./基础设定/TOOLS.md

### 交接段（复杂任务必填，简单任务可省）
- 完成后通知谁：{花名/Seele}
- 下游任务：{描述}
```

**修改文件**：SOUL.md（在「任务分配标准」后追加）

### P2. 上下文预算意识

**问题**：主 session 没有量化感知，大段日志和代码 diff 是上下文杀手，不知道什么时候会爆。

**方案**：建立上下文预算规则，写入 SOUL.md（行为约束）和 TOOLS.md（经验参考）。

**SOUL.md 追加规则**：
```markdown
## 上下文预算规则

### 预算感知
- 每次会话约 100-120K tokens，截断通常发生在 80% 处
- 1 个工具调用（含返回）≈ 1-3K tokens
- 1 个 read_file 100行 ≈ 3-5K tokens
- 1 段完整编译日志（200行）≈ 8-15K tokens
- 1 个代码 diff（500行）≈ 10-20K tokens

### 节流策略
1. **累计超过 50K tokens 内容时**，停止自己读大文件，改 spawn sub-agent 处理
2. **单次 read_file 超 100 行时**，用 offset/limit 分批读，或交给 sub-agent
3. **用户贴长日志时**，只提取 Error 行 + 上下文 5 行（见 P3），不全文引用
4. **代码 diff 超 50 行时**，只看签名和变更摘要，细节交给 sub-agent
5. **接近截断征兆**：响应变慢、工具返回不完整 → 立即 spawn 并写 MEMORY.md 保存状态

### 预算自检
- 每完成 1 个大任务（或每 5 次工具调用），心里默算：已消耗约多少？
- 超过预算 60% 时，主动精简后续操作：少读、多派、快写
```

**TOOLS.md 追加经验**：
```markdown
## 上下文管理经验
- 主session上下文约100-120K tokens，截断线在~80%
- 大文件(>5KB)不要在主session读全文，spawn sub-agent处理
- 编译日志只提取Error行，全文写文件不贴对话
- 代码diff超50行只看摘要，细节让sub-agent处理
- 接近截断时优先写MEMORY.md保存状态，再派sub-agent继续
```

### P3. 日志处理强制规则

**问题**：用户提醒过"只贴Error行"但没写成强制规则，用户粘贴整段日志会快速消耗上下文。

**方案**：在 SOUL.md 新增强制规则，任何情况下必须遵守。

**SOUL.md 追加规则**：
```markdown
## 日志处理铁律

### 强制规则（不可违反）
1. **用户贴日志时，只提取 Error 行 + 每个错误前后各 5 行上下文**
2. **绝不在对话中引用完整日志**，全文写入文件后只给路径
3. **sub-agent 处理编译错误时**，汇报格式："N 个错误，关键：{前3个错误的摘要}"
4. **写文件时可以保留完整日志**，但对话中只放摘要+路径
5. **违者等于浪费团队上下文预算**，这是集体资源

### 日志摘要模板
```
编译结果：{N} Error, {M} Warning
关键错误：
1. {文件}:{行号} - {错误摘要}
2. {文件}:{行号} - {错误摘要}
3. {文件}:{行号} - {错误摘要}
...如超过3个，其余见完整日志文件
完整日志：{文件路径}
```
```

**修改文件**：SOUL.md（在「上下文预算规则」后追加）

### P4. Sub-agent 产出验收流程

**问题**：sub-agent 改完代码直接转给用户，没过审核，质量问题无法拦截。

**方案**：建立分层验收流程，写入 SOUL.md。

**SOUL.md 追加规则**：
```markdown
## Sub-agent 产出验收流程

### 分层标准
| 层级 | 类型 | 验收要求 | 示例 |
|------|------|----------|------|
| Tier 1 | 代码变更/架构决策/用户交付物 | **必须 Rex 审核** | 代码修复、SDK改动、文档发布 |
| Tier 2 | 内部文档/日志/非关键更新 | Seele 自检即可 | dev-log、调试记录 |
| Tier 3 | 极简任务(<10行/纯查询) | 直接交付 | 查文件、读配置 |

### Tier 1 验收流程
1. Eli/代码sub-agent 完成任务 → 报告 Seele
2. Seele **不直接转交用户**，spawn Rex 审核产出
3. Rex 审核清单：
   - [ ] 功能正确性：是否解决了目标问题
   - [ ] 无回归：是否破坏了已有功能
   - [ ] 规范合规：命名/格式/架构符合项目约定
   - [ ] 产出完整：该改的都改了，没遗漏
4. Rex 通过 → Seele 交付用户；Rex 不通过 → 带具体意见退回原 sub-agent

### 快速通道（紧急情况）
- 编译错误修复可跳过 Rex 审核，但 Seele 必须自检：
  - 改动范围是否最小
  - 是否引入新编译错误
  - 是否符合已有模式（变量命名、include顺序等）
- 跳过审核后必须补审，在 dev-log 中标注「跳过审核-紧急」

### Sub-agent 汇报格式
完成任务后必须包含：
1. **做了什么**（1-3句）
2. **改了哪些文件**（路径列表）
3. **自检结果**（通过/问题）
4. **需要后续关注**（如有）
```

**修改文件**：SOUL.md（在「日志处理铁律」后追加）

### P5. 心跳巡检误报修复

**问题**：心跳巡检报 TOOLS.md 不存在是假警报。巡检逻辑太粗糙（可能读了错误路径，或用了 bash `test -f` 而文件在特殊挂载点）。

**方案**：修改心跳日程描述，明确巡检验证方式和路径。

**当前问题分析**：
- `./基础设定/TOOLS.md` 文件存在（read_file 可读），但 bash `ls` 和 `test -f` 看不到
- 说明该文件在沙箱的文件系统中通过 read_file 工具可访问，但 bash 命令行路径不同
- 心跳巡检如果用 bash 检查文件存在性，会误报

**修复方案**：
心跳巡检描述中，文件存在性检查改为：
1. 使用 `read_file` 工具（而非 bash `test -f` / `ls`）验证文件是否可读
2. 每个检查项明确指定验证方式
3. 对于只读检查的文件，只要 read_file 返回内容即视为存在

**心跳日程描述（更新版）**：
```
Seele铁律巡检（每2小时）：
1. 读 SOUL.md — 用read_file读./基础设定/SOUL.md，确认「截断恢复必读」三件事存在
2. 读 TOOLS.md — 用read_file读./基础设定/TOOLS.md，确认「截断恢复行为守则」段落存在
3. 读 MEMORY.md — 用read_file读./MEMORY.md，确认待办列表非空
4. 读 SECRET.md — 用read_file读./SECRET.md，确认非空（不输出内容）
5. 检查目录 — 用bash检查./Neshama/dev-logs/和./Neshama/dev-plans/目录存在
注意：文件存在性验证必须用read_file，不要用bash的test -f或ls，部分文件在沙箱特殊挂载点bash无法访问
```

**修改方式**：通过 calendar_update 更新日程描述（ID: eec229d9-84a1-41f5-a377-fd152e0148b5）
