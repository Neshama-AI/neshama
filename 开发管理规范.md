# Neshama 开发管理规范

## 文件结构

```
./Neshama/
├── dev-plans/           # 开发计划（每轮开始前写）
│   ├── 2026-05-08-unity-sdk.md
│   ├── 2026-05-08-ue5-sdk.md
│   └── ...
├── dev-logs/            # 开发日志（每轮结束后写）
│   ├── 2026-05-08-unity-sdk.md
│   ├── 2026-05-08-ue5-sdk.md
│   └── ...
├── debug-plans/         # 调试计划（遇到编译/运行问题时写）
│   ├── 2026-05-08-unity-6rounds.md
│   └── ...
├── debug-logs/          # 调试日志（每个问题修复后记录）
│   ├── 2026-05-08-unity-6rounds.md
│   └── ...
└── retros/              # 阶段性回顾（整合日志+提炼经验）
    ├── 2026-05-W1.md
    └── ...
```

## 写作规则

### 开发计划 (dev-plans/)
- **何时写**：接到任务后、动手之前
- **格式**：
  1. 目标（一句话）
  2. 预估步骤
  3. 已知风险
  4. 验收标准
- **命名**：`{日期}-{任务简称}.md`

### 开发日志 (dev-logs/)
- **何时写**：任务完成或中断时
- **格式**：
  1. 实际做了什么
  2. 产出文件/commit
  3. 遗留问题
  4. 下次接续点
- **命名**：同开发计划

### 调试计划 (debug-plans/)
- **何时写**：遇到编译错误/运行异常，开始排查前
- **格式**：
  1. 问题描述
  2. 已知线索
  3. 排查步骤（按优先级）
  4. 预期结果

### 调试日志 (debug-logs/)
- **何时写**：每个问题修复后立即记录
- **格式**：
  1. 错误现象
  2. 根因
  3. 修复方案
  4. 踩坑点（给未来的自己）
- **关键**：记录中间试过但失败的方案，避免重复

### 阶段回顾 (retros/)
- **何时写**：每周/每阶段结束
- **格式**：
  1. 本阶段完成项
  2. 经验教训提炼
  3. 工具/流程改进
  4. 下阶段规划

## 流程

```
接任务 → 写dev-plan → 执行 → 写dev-log
                              ↓
          遇问题 → 写debug-plan → 排查 → 写debug-log → 继续执行
                                                    ↓
                              阶段结束 → 写retro → 更新MEMORY.md
```

## 与MEMORY.md的关系
- MEMORY.md：全局状态（项目进度、待办、关键决策），只写结论
- dev-plans/logs：过程记录，写详细步骤
- debug-plans/logs：问题记录，写排查过程
- retro：提炼结论回写MEMORY.md和TOOLS.md

## 日志生命周期

临时日志（dev-logs/debug-logs）→ 整合进完整工作日志 → 临时日志完成使命

### 完整工作日志
- **路径**：`./Neshama/dev-logs/{日期}-完整.md`（当日所有任务的汇总）
- **整合时机**：每日结束时或阶段性切换时
- **内容**：从当日所有临时日志中提取，合并去重，按时间线组织

### 上下文复用
- 分身接任务时，只需读完整日志+要点总结，不用读所有临时日志
- 主对话只需要结论，不需要过程细节
- 流程：`分身读完整日志 → 提炼要点 → 告诉主对话 → 主对话更新MEMORY.md`

### 示例
```
临时：dev-logs/2026-05-08-unity-sdk.md
临时：dev-logs/2026-05-08-ue5-sdk.md
整合→：dev-logs/2026-05-08-完整.md
要点→：MEMORY.md 中的待办/完成状态
```
