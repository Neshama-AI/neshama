## 1. 类型重复定义（最致命）✅ 已修复

### 问题描述
`NeshamaTypes.h` 和 `SoulEngine/` 下多个头文件定义了同名类型，导致 ODR 违反：
- `EEmotionType`：NeshamaTypes.h 有 Shame 无 Neutral，EmotionTypes.h 有 Neutral 无 Shame
- `EGameEventType`：完全不同的枚举值
- `FEmotionState`：NeshamaTypes.h 用 TMap，EmotionTypes.h 用独立 float
- `FGameEvent`：结构完全不同
- `EBehaviorType`：NeshamaTypes.h 和 BehaviorTypes.h 都定义了

### 修复方案
**SoulEngine 内部类型加 `Soul` 前缀，两套类型各自独立不冲突：**

| 原类型名 | SoulEngine 新名 | 所在文件 |
|----------|-----------------|----------|
| `EEmotionType` | `ESoulEmotionType` | `SoulEngine/Emotion/EmotionTypes.h` |
| `EGameEventType` | `ESoulEventType` | `SoulEngine/Emotion/EmotionTypes.h` |
| `FEmotionState` | `FSoulEmotionState` | `SoulEngine/Emotion/EmotionTypes.h` |
| `FGameEvent` | `FSoulGameEvent` | `SoulEngine/Emotion/EmotionTypes.h` |
| `EBehaviorType` | `ESoulBehaviorType` | `SoulEngine/Behavior/BehaviorTypes.h` |

**补充枚举值：**
- `ESoulEmotionType`（SoulEngine 版）增加 `Shame` 值
- `EEmotionType`（NeshamaTypes.h 版）增加 `Neutral` 值

**批量更新了以下 SoulEngine 文件中的所有引用：**
- `BehaviorMapper.h/.cpp` — `EBehaviorType` → `ESoulBehaviorType`
- `CompositeEmotion.h/.cpp` — `EEmotionType` → `ESoulEmotionType`
- `EmotionEngine.h/.cpp` — `EEmotionType` → `ESoulEmotionType`, `EGameEventType` → `ESoulEventType`, `FEmotionState` → `FSoulEmotionState`
- `EmotionTypes.h/.cpp` — 全部重命名
- `EventMappings.h/.cpp` — `EGameEventType` → `ESoulEventType`, `EEmotionType` → `ESoulEmotionType`
- `GameEventProcessor.h/.cpp` — `EGameEventType` → `ESoulEventType`, `FGameEvent` → `FSoulGameEvent`
- `MemorySystem.h/.cpp` — `EGameEventType` → `ESoulEventType`
- `PersonalityEvolver.h/.cpp` — `FEmotionState` → `FSoulEmotionState`

## 3. BlueprintConstructor 不是合法 UFUNCTION ✅ 已修复

### 问题描述
`NeshamaClient.h` 构造函数上使用了 `UFUNCTION(BlueprintConstructor)` 宏，UE5 不支持。

### 修复方案
删除构造函数上的 `UFUNCTION(BlueprintConstructor, Category = "Neshama|Client")` 宏，保留普通构造函数声明。

---

## 4. TMap 不能做 UFUNCTION 参数 ✅ 已修复

### 问题描述
`NPCSoulComponent.h` 和 `NeshamaClient.h` 中 `SendGameEvent` 方法的 `TMap<FString, FString>` 参数不能直接暴露给 Blueprint。

### 修复方案
在两个文件的 `SendGameEvent` UFUNCTION 宏中添加 `meta = (BlueprintInternalUseOnly = "true")`，使该函数仅在 C++ 层面可用，不暴露给 Blueprint。

- `NeshamaClient.h`: `UFUNCTION(BlueprintCallable, Category = "Neshama|Client|Event", meta = (BlueprintInternalUseOnly = "true"))`
- `NPCSoulComponent.h`: `meta = (DisplayName = "Send Game Event", ToolTip = "...", BlueprintInternalUseOnly = "true")`

## 设计决策说明

### 为什么选择"加前缀"而非"合并类型"？
1. **最小侵入性**：NeshamaTypes.h 是面向外部开发者的公共 API，改动它会影响所有使用者
2. **职责分离**：SoulEngine 内部类型（8 个独立 float 字段）和 SDK 公共类型（TMap 动态结构）用途不同
3. **向后兼容**：NeshamaTypes.h 的 FEmotionState/TMap 结构可以灵活序列化 JSON，而 SoulEngine 的 FSoulEmotionState/独立 float 结构适合高性能计算
4. **避免级联修改**：如果合并类型，需要修改所有 Blueprint 节点和序列化逻辑

### EEmotionType 两边都保留了
- **NeshamaTypes.h 版**：9 基础情绪 + Shame + Neutral = 11 值，面向外部 API
- **SoulEngine 版 (ESoulEmotionType)**：8 基础 + Shame + Neutral = 10 值，面向内部计算
- 两者命名不同（`EEmotionType` vs `ESoulEmotionType`），不存在冲突
